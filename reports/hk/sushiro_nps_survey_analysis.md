# 壽司郎問卷調查系統 (NPS) 香港版 填寫紀錄分析報告

本報告根據 `reports/hk/nps.sushiro.com.hk.har` 與 `reports/hk/har_export_1783697212340.json` 的網路填寫紀錄，整理出香港版問卷的系統資訊、API 串接規格，以及完整的問卷問題清單與回答選項。

## 1. 系統與認證資訊

- **問卷入口網址**: [https://nps.sushiro.com.hk/](https://nps.sushiro.com.hk/)
- **API 基底網址 (Base URL)**: `window.location.protocol + "//" + window.location.host` (即同源發送)
- **API 認證機制**:
  使用 Bearer Token 進行身分驗證。其完整取得與發送機制如下：

  ### A. 憑證來源 (Hardcoded APIKEY)
  授權金鑰 `APIKEY` 硬編碼在香港版首頁 HTML 中，做為全域常數宣告：
  ```html
  <script>
    const APIKEY = "web:1c263ca9a5b36e58b8a3f3ca02f54f97";
  </script>
  ```
  * **金鑰格式**：`web:<32位元MD5雜湊值>`。香港版與台灣版的 APIKEY 不同，香港版專屬 Key 為 `web:1c263ca9a5b36e58b8a3f3ca02f54f97`。

  ### B. 前端組裝與發送方式
  與台灣版結構一致，在主邏輯腳本中將變數拼接為 `Authorization: Bearer <APIKEY>`：
  ```javascript
  let headers = {
      headers: {
          'Content-Type': 'application/json',
          'Authorization': 'Bearer ' + APIKEY
      }
  };
  ```
  並隨 Axios 發送 POST 請求至：
  - **啟動問卷**：`/api/v1/surveys/start`
  - **提交下一步**：`/api/v1/surveys/next`

  ### C. HAR 檔案中缺乏此 Header 的技術原因
  - **瀏覽器導出 HAR 檔時的安全隱私過濾**：與台灣版相同，瀏覽器在導出 HAR 檔案時，預設會剔除包含在 headers 裡的 `Authorization` 與 `Cookie` 等敏感資訊。

## 3. API 串接規格

問卷系統採用 **Stateless (無狀態)** 設計。前端在回答問題並跳至下一步時，必須在 Request Payload 的 `answers` 與 `comments` 中**帶上之前所有步驟已填寫的完整答案**。伺服器會根據目前已收到的答案，於 Response 中返回「下一個問題頁面」的結構。

### A. 啟動問卷 API

- **Endpoint**: `/api/v1/surveys/start`
- **Method**: `POST`
- **Request Payload**:
  ```json
  {
    "invitation_code": "26071010",
    "total_price": "200",
    "visited_time": "3"
  }
  ```
- **Response Payload (部分簡化)**:
  ```json
  {
    "status": 200,
    "message": "OK",
    "data": {
      "status": 0,
      "progress": 0,
      "mst_question_page": {
        "id": 1,
        "name": "総合満足度",
        "display_title": "就關於今次光臨的壽司郎店舖，請回答下列問題。 ",
        "display_order": 1,
        "mst_questions": [
          {
            "id": 1,
            "no": 1,
            "content": "請您對此店舖的整體滿意度評分。",
            "form_type": 1,
            "is_required": true,
            "options": [
              { "no": 1, "content": "非常滿意" },
              { "no": 2, "content": "滿意" }
            ]
          }
        ]
      }
    }
  }
  ```

### B. 下一步 / 提交回答 API

- **Endpoint**: `/api/v1/surveys/next`
- **Method**: `POST`
- **Request Payload**:
  ```json
  {
    "invitation_code": "26071010",
    "total_price": "200",
    "visited_time": "3",
    "answers": [
      {
        "mst_question_id": 1,
        "answered_option_no": "1"
      }
    ],
    "comments": []
  }
  ```
- **Response Payload (一般步驟)**:
  若問卷尚未結束，返回下一頁的問題結構：
  ```json
  {
    "status": 200,
    "message": "OK",
    "data": {
      "status": 0,
      "progress": 7,
      "mst_question_page": {
        "id": 5,
        "name": "關鍵性要素設問",
        "display_title": "請根據您的滿意度填寫下列各項（最高5分）",
        "mst_questions": [...]
      }
    }
  }
  ```
- **Response Payload (最後步驟 - 完成問卷)**:
  當所有問題填寫完畢，伺服器判定結束後返回折價券認證碼：
  ```json
  {
    "status": 200,
    "message": "OK",
    "data": {
      "status": -1,
      "progress": 100,
      "code": "92694" // 5 位數折價券認證碼
    }
  }
  ```

## 5. 問卷系統安全性與繞過測試 (PoC)

我們使用香港版專屬 APIKEY `web:1c263ca9a5b36e58b8a3f3ca02f54f97` 進行了安全性繞過測試，結果與發現如下：

### A. 測試方法與結果

測試直接向 `/api/v1/surveys/next` 提交包含全部 26 個問題答案的 Payload，跳過所有換頁步驟：

1. **情境一：使用有效邀請碼 `26071010` 一次提交全部答案**
   * **API 回應狀態**：`200 OK`
   * **Response Body**：
     ```json
     {
       "status": 200,
       "message": "OK",
       "data": {
         "status": -1,
         "progress": 100,
         "code": "92694"
       }
     }
     ```
   * **結果**：繞過成功，直接取得 5 位數折價券認證碼 `92694`。

2. **情境二：使用 6 日內有效邀請碼 `26070901` 一次提交全部答案**
   * **API 回應狀態**：`200 OK`
   * **Response Body**：
     ```json
     {
       "status": 200,
       "message": "OK",
       "data": {
         "status": -1,
         "progress": 100,
         "code": "92694"
       }
     }
     ```
   * **結果**：同樣成功繞過，且獲得相同的認證碼 `92694`。

3. **情境三：使用過期邀請碼 `12878444` 一次提交全部答案**
   * **結果**：`400 Bad Request`，提示過期錯誤（用餐日期 6 日之內有效限制）。

---

### B. 關鍵安全性結論

1. **無狀態與步驟繞過**：後端並未在伺服器端追蹤答題步驟軌跡，只需發送一次填滿 26 題答案的 POST 請求，即可獲取折扣碼。
2. **日期格式與 6 日限制**：`invitation_code` 同樣採 `YYMMDDSS` 格式檢驗用餐日期是否在 6 日內。
3. **認證碼靜態常數**：在該檔期內，香港版的所有成功填寫者獲得的認證碼皆為固定的常數 `"92694"`。
