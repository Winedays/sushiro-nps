# 壽司郎問卷調查系統 (NPS) 填寫紀錄分析報告

本報告根據 `nps.sushiro.com.tw.har` 的網路填寫紀錄，整理出壽司郎問卷的系統資訊、API 串接規格，以及完整的問卷問題清單與回答選項。

---

## 1. 系統與認證資訊

- **問卷入口網址**: [https://nps.sushiro.com.tw/](https://nps.sushiro.com.tw/)
- **API 基底網址 (Base URL)**: `window.location.protocol + "//" + window.location.host` (即同源發送)
- **API 認證機制**:
  使用 Bearer Token 進行身分驗證。其完整取得與發送機制如下：

  ### A. 憑證來源 (Hardcoded APIKEY)
  授權金鑰 `APIKEY` 並非透過網路 API 動態取得，而是**直接硬編碼（Hardcoded）**在問卷首頁的 HTML 原始碼中，作為全域常數宣告：
  ```html
  <script>
    const APIKEY = "web:624a677331626b5044e9bb25a1fcf8a9";
  </script>
  ```
  * **金鑰格式**：`web:<32位元MD5雜湊值>`。其中 `624a677331626b5044e9bb25a1fcf8a9` 具有標準 MD5 雜湊的特徵。

  ### B. 前端組裝與發送方式
  在前端主邏輯腳本 [/js/user/user.js](file:///mnt/c/adlerau/sushiro_nps/nps.sushiro.com.tw.har)（第 87~92 行）中，前端會讀取上述全域變數，並將其組裝為 Axios 請求的 headers 配置：
  ```javascript
  let headers = {
      headers: {
          'Content-Type': 'application/json',
          'Authorization': 'Bearer ' + APIKEY
      }
  };
  ```
  當用戶在前端填寫資料並點擊按鈕時，前端會將 `headers` 當作第三個參數（config）傳入 Axios 的 POST 請求中以呼叫後端 API：
  - **啟動問卷**：`instance.post(URL + "/api/v1/surveys/start", { ... }, headers)`
  - **提交下一步**：`instance.post(URL + "/api/v1/surveys/next", data, headers)`

  ### C. HAR 檔案中缺乏此 Header 的技術原因
  在 HAR 紀錄的實際請求標頭（Request Headers）中，並未發現 `Authorization` 欄位。原因主要為：
  - **瀏覽器導出 HAR 檔時的安全隱私過濾**：現代瀏覽器（如 Chrome, Edge）在導出 HAR 檔時，出於安全性與隱私防護，**預設會過濾並清除所有敏感的驗證資訊**（包括 `Authorization` 標頭、`Cookie` 等），除非在匯出時有手動勾選 "Include information about cookies and authentication headers in the HAR file"。

## 3. API 串接規格

問卷系統採用 **Stateless (無狀態)** 設計。前端在回答問題並跳至下一步時，必須在 Request Payload 的 `answers` 與 `comments` 中**帶上之前所有步驟已填寫的完整答案**。伺服器會根據目前已收到的答案，於 Response 中返回「下一個問題頁面」的結構。

### A. 啟動問卷 API

- **Endpoint**: `/api/v1/surveys/start`
- **Method**: `POST`
- **Request Payload**:
  ```json
  {
    "invitation_code": "26070508",
    "total_price": "130",
    "visited_time": "3"
  }
  ```
- **Response Payload (部分簡化)**:
  ```json
  {
    "status": 200,
    "message": "OK",
    "data": {
      "status": 0,
      "progress": 0,
      "mst_question_page": {
        "id": 1,
        "name": "綜合滿意度",
        "display_title": "請針對本次蒞臨消費的店舖回答以下問題。",
        "display_order": 1,
        "mst_questions": [
          {
            "id": 1,
            "no": 1,
            "mst_question_page_id": 1,
            "content": "對於本次蒞臨消費的店舖之整體滿意度",
            "form_type": 1,
            "is_required": true,
            "options": [
              { "no": 1, "content": "非常滿意" },
              { "no": 2, "content": "滿意" }
            ]
          }
        ]
      }
    }
  }
  ```

### B. 下一步 / 提交回答 API

- **Endpoint**: `/api/v1/surveys/next`
- **Method**: `POST`
- **Request Payload**:
  ```json
  {
    "invitation_code": "26070508",
    "total_price": "130",
    "visited_time": "3",
    "answers": [
      {
        "mst_question_id": 1,
        "answered_option_no": "2"
      },
      {
        "mst_question_id": 4,
        "answered_option_no": "1"
      }
    ],
    "comments": [
      {
        "mst_question_id": 72,
        "answered_text": "沒有"
      }
    ]
  }
  ```
- **Response Payload (一般步驟)**:
  若問卷尚未結束，返回下一頁溫問題結構：
  ```json
  {
    "status": 200,
    "message": "OK",
    "data": {
      "status": 0,
      "progress": 25,
      "mst_question_page": {
        "id": 5,
        "name": "關鍵性要素設問",
        "display_title": "請根據您的滿意度來填寫以下的問題（每個滿意度最高五分）",
        "mst_questions": [...]
      }
    }
  }
  ```
- **Response Payload (最後步驟 - 完成問卷)**:
  當所有問題填寫完畢，伺服器判定結束後返回折價券認證碼：
  ```json
  {
    "status": 200,
    "message": "OK",
    "data": {
      "status": -1,
      "progress": 100,
      "code": "70039" // 5 位數折價券認證碼，用以填寫於紙本折價券中
    }
  }
  ```

## 5. 問卷系統安全性與繞過測試 (PoC)

為了評估問卷系統的健壯性，我們進行了繞過問卷填寫步驟的實作測試（Proof of Concept, PoC），測試結果與關鍵安全性發現如下：

### A. 測試方法與結果

測試透過直接向 `/api/v1/surveys/next` 發送一個包含**所有 23 個問題完整答案**的 POST 請求，跳過中間所有單步提交的流程。

1. **情境一：使用原邀請碼 `26070508` 一次提交全部答案**
   * **API 回應狀態**：`200 OK`
   * **Response Body**：
     ```json
     {
       "status": 200,
       "message": "OK",
       "data": {
         "status": -1,
         "progress": 100,
         "code": "70039"
       }
     }
     ```
   * **結果**：直接繞過步驟成功，且成功取得 5 位數折價券認證碼 `70039`。

2. **情境二：使用符合 6 日內規則之新型邀請碼 `26070901` 一次提交全部答案**
   * **API 回應狀態**：`200 OK`
   * **Response Body**：
     ```json
     {
       "status": 200,
       "message": "OK",
       "data": {
         "status": -1,
         "progress": 100,
         "code": "70039"
       }
     }
     ```
   * **結果**：同樣成功繞過，且取得與情境一完全相同的認證碼 `70039`。

3. **情境三：使用過期或隨機格式邀請碼 `12878444` 一次提交全部答案**
   * **API 回應狀態**：`400 Bad Request`
   * **Response Body**：
     ```json
     {
       "status": 400,
       "message": "Bad Request",
       "data": {
         "invitation_code": {
           "expired": "問券調查需於用餐當日起6日內填寫, 已超過問券可填寫時間",
           "valid": "認證碼錯誤"
         }
       }
     }
     ```
   * **結果**：因邀請碼的日期校驗不通過，被後端伺服器阻擋。

---

### B. 關鍵安全性結論

1. **無步驟順序檢驗 (No State/Session Track)**
   問卷伺服器後端完全**沒有**對答題步驟進行狀態追蹤或防重複校驗。只要發送的 Payload 包含全部 23 個問題的答案，伺服器就會直接判定問卷填寫完成，進而發放認證碼。這使得自動化指令可以直接一次性發送答案，繞過前端十多次的 UI 換頁操作。

2. **邀請碼日期檢驗機制 (YYMMDDSS 格式)**
   後端會解析 `invitation_code` 的前 6 碼（如 `260709` 代表 2026 年 7 月 9 日），並校驗此用餐日期是否在當前伺服器時間的 **6 天之內**。若符合 6 日內之限制則視為有效，若逾期則回傳錯誤。

3. **折價券認證碼靜態化 (Static Promo Code)**
   在所有成功繞過並完成問卷的測試中，後端回傳的 `code` 均為固定值 `"70039"`。這代表系統的折價券認證碼並非動態生成或一次性（one-time-use）使用的隨機金鑰，而是在此檔期內或特定規則下**完全固定不變的靜態常數**。
